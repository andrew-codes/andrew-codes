I am a home automation enthusiast and love building and learning new things. My smart home is already configured with all sorts of fun and useful automations, but I wanted to build the missing link: a voice assistant
tailored to how I actually live.

This article is a wider versus deeper look at the system. Before diving into the design, let us start with what it looks like in action.

## Why Build My Own?

The obvious question is: why not just use Alexa or Google Home?

Privacy is an immediate benefit of running your own solution, though it was not my primary motivation. The bigger driver is that commercial assistants are fundamentally general-purpose tools, and that generality comes
with real limits on customization. Building my own means I can shape the assistant around how I actually live rather than working around what a platform allows.

A few concrete examples of things my assistant does that would be difficult or impossible with off-the-shelf alternatives:

- **Grocery list integration**: Adding items to a shopping list that surfaces in home dashboards and triggers a reminder when we arrive at the relevant store — not an Alexa or Google task list.
- **Spotify with room awareness**: Playing music and targeting specific rooms or grouping them for whole-home audio, with full playback control.
- **Guest concierge**: A self-service experience that helps guests understand how to control the home without needing to ask.

There is also the honest answer: this is a hobby I enjoy. I learn an enormous amount from the process of building, not just from the end result.

## Goals

Let us outline a few scenarios I want this assistant to handle.

| Scenario             | Goal                                                                                                                        |
| :------------------- | :-------------------------------------------------------------------------------------------------------------------------- |
| Weather              | Help me make choices when I am getting ready, so I am wearing weather-appropriate clothing.                                 |
| Set timers           | One or more timers can be set at a time.                                                                                    |
| Guest wifi           | Guests may not have the wifi password. Show the wifi pass phrase for the guest network.                                     |
| Party mode           | Play, pause, skip, and stop music. Show cover art for the currently playing song. I will tell it if the volume is too loud. |
| Chill out            | Change the temperature or HVAC mode when the room is too warm.                                                              |
| Locks                | Tell me if doors are locked or unlocked. Assistants can lock, but should not unlock exterior doors.                         |
| Batteries            | Let me know if any smart devices need battery replacements.                                                                 |
| Lights, fans, action | Control lights, fans, and other devices in an intuitive way.                                                                |
| Lists and groceries  | Add to-do items to my task list and groceries to my shopping list.                                                          |

## The Setup

Here is an overview of the major components that make this work.

- **Kotlin-based native Android app**<br/> Runs on a tablet and handles wake word detection, response visualization, and screen sleep and wake behavior. Visualization and settings are controlled through Home Assistant
  via a Python integration. Using [ViewAssistant Companion app](https://github.com/msp1974/ViewAssistCompanionApp).<br/><br/> The device must be Android. iOS does not allow third-party apps to continuously listen in the
  background, which is a hard requirement for wake word detection. A browser-based UI has the same problem — it cannot always be on and listening. I also considered repurposing an older Echo Show, since specific firmware
  versions can be rooted to run custom software. In practice, finding a device running the right exploitable version is unreliable — most sellers do not know what firmware version they have — and the approach is not
  sustainable long-term since it depends on exploits that can be patched at any time.

- **AI workloads**<br/> Speech to text uses [Faster Whisper](https://github.com/SYSTRAN/faster-whisper) and text to speech uses [Piper](https://github.com/rhasspy/piper), both running in Kubernetes. A single NVIDIA GPU
  is shared through time-sliced resource constraints defined in deployment YAML.<br/><br/> I had a spare GPU, which made self-hosted AI workloads a natural fit. My initial setup ran both services on a single Debian VM,
  which worked but was harder to maintain and update than a GitOps-based approach. Moving to Kubernetes meant learning several things in sequence: passing through the GPU at the hypervisor layer via PCIe, making the GPU
  visible to the Kubernetes cluster at the node level, and then distributing that single GPU resource across more than one pod. I also explored MIG (Multi-Instance GPU) along the way before discovering my GPU does not
  support it. Piper is a well-regarded choice in the Home Assistant community. Faster Whisper had strong performance relative to the alternatives I evaluated at the time, though I did not do an exhaustive comparison.

- **Home Assistant**<br/> Connects the Android app to AI workloads via the Wyoming protocol.<br/><br/> I was already running Home Assistant before starting this project, having previously tried SmartThings, OpenHAB, and
  a custom Crestron solution. Home Assistant strikes the right balance: broad device and service compatibility, but still flexible enough for a maker who wants to go deeper than the defaults.

- **Python-based Home Assistant integrations**<br/> Control Android app settings and connect Home Assistant Assist to OpenAI. Using
  [Extended OpenAI Conversation](https://github.com/jekalmin/extended_openai_conversation).<br/><br/> I started building my own HACS integration for OpenAI, but while I was working on it someone else published one that
  covered the same ground. I ended up using theirs, and it only supports OpenAI. I tried local LLMs as an alternative, but found them unreliable when the entire assistant is driven by a single large prompt covering many
  distinct areas (music, thermostat, timers, etc.). The cost-to-performance ratio of OpenAI made it a reasonable pragmatic choice for now. My next iteration will revisit local models, but with a different architecture: a
  LangChain agent per capability area rather than one monolithic prompt. That approach should give a local LLM a much better chance of handling each scenario reliably, but will also require building a new custom Python
  integration to replace the current one.

- **Playnite Web**<br/> A custom web experience for browsing and remote controlling a game library across PC, PlayStation, Xbox, and Nintendo. This is the default view for the game room tablet. See
  [this repo](https://public.home.playniteweb.com/).

- **Kubernetes**<br/> All AI workloads run as pods in an existing cluster managed with Flux GitOps.<br/><br/> I already had the cluster running, so deploying here was a natural extension of existing infrastructure.
  GitOps makes it straightforward to upgrade or roll back service versions without touching the underlying VM's operating system. It also turned out to be a great opportunity to go deeper on GPU resource sharing in
  Kubernetes, which I would not have encountered otherwise.

### Architecture Snapshot

At a high level, the user speaks to the Android tablet, wake word detection hands off to Home Assistant Assist, and Home Assistant routes speech-to-text and text-to-speech through AI services running in Kubernetes.
Responses are then visualized and played back on the tablet, with Home Assistant coordinating automations, tools, and external APIs.

## Challenges and Next Steps

The biggest challenge is tuning the AI prompt. There are several discrete sets of functionality (weather, music, timers, etc.) that all need to be reliably triggered by user speech. What I'm finding is that AI can
struggle to consistently trigger the right functionality, especially when user speech is ambiguous or contains multiple intents. For example, "Set a timer for 10 minutes and play some music" contains two distinct actions
that need to be parsed and executed correctly.

To address this, I am experimenting with a few strategies:

- **Prompt engineering**:<br/> Iteratively refining the AI prompt to provide clearer instructions and examples for handling multiple intents in a single user utterance.

- **Intent classification via langchain agent**:<br/> Implementing a separate, custom intent classification [agent](https://www.langchain.com/) that can first categorize user speech into distinct intents before passing
  it to the main AI for execution. This could help ensure that each intent is handled appropriately, even when multiple intents are present.

## Conclusion

Building a custom voice assistant for my smart home has been a rewarding project that combines AI, home automation, and user experience design. Huge shout out to the open source community for providing the tools and
inspiration to make this possible. While there are challenges to overcome, especially around reliably parsing user intent, the system is already providing value and fun interactions in my daily life. I look forward to
continuing to refine the AI prompt and exploring additional features and integrations in the future.
